# 베르셀 배포

## 목적

- 지금 단계에서는 도메인 연결 없이 `Vercel`만 먼저 배포한다.
- 배포 주소는 기본 `*.vercel.app` 도메인을 사용한다.
- 추후 가비아 도메인과 Cloudflare는 별도 작업으로 연결한다.

## 배포 전 상태

- `apps/web`는 **데이터·CMS·파일(이미지) 저장은 Supabase** 기준이다.
- **Firebase는 로그인·회원·카카오 연동과, 서버에서 그 토큰을 검증하는 용도**로만 쓴다. (스토어 이미지 URL은 Supabase Storage + `cms` 테이블.)

## 배포 순서

1. 현재 저장소를 GitHub에 push
2. Vercel에서 `New Project`
3. GitHub 저장소 import
4. 프로젝트 설정
   - Framework Preset: `Next.js`
   - Root Directory: `apps/web`
5. Environment Variables 입력
6. Deploy

## Root Directory

**Vercel → Settings → General → Node.js Version 은 20.x 권장** (`package.json` 의 `engines.node` 와 맞춤).

**Vercel 프로젝트의 Root Directory는 저장소 루트(`.`)** 로 둔다.

**주의:** Root Directory를 `apps/web`으로 두면 `pnpm install`이 하위 폴더에서 돌아가 루트 `node_modules`·`pnpm-lock.yaml`과 어긋나 **lockfile 무시 오류**가 날 수 있다. 반드시 **저장소 루트**로 두고, 루트의 `vercel.json`이 빌드를 잡게 한다. (`installCommand`는 `git rev-parse --show-toplevel`로 루트에 맞추지만, 대시보드 Root가 `apps/web`이면 `buildCommand`의 `cd apps/web`과 겹쳐 깨질 수 있음.)

**배포가 Error로 끝난 뒤 `Redeploy`가 안 될 때:** Vercel이 *This deployment can not be redeployed. Please try again from a fresh commit* 라고 하면, 실패한 배포는 같은 방식으로 다시 못 돌리는 경우가 있다. **새 커밋을 푸시**하거나 상단 **Deployments → ⋯ → Create Deployment** 로 `master` 최신을 한 번 배포한다.

**`apps/web/apps/web/.next` 를 찾을 수 없다 (Output Directory 오류):** Vercel **Settings → General → Root Directory** 가 **`apps/web`** 이면, 출력물은 이미 그 아래에 생기므로 `vercel.json` 의 `outputDirectory` 는 **`.next`** 만 써야 한다. **`apps/web/.next` 를 넣으면 경로가 두 번 붙어** 위 오류가 난다.  
반대로 Root Directory를 **저장소 루트 `.`** 로 둔다면 `outputDirectory` 는 **`apps/web/.next`** 로 두어야 한다. (지금 저장소 `vercel.json` 은 **Root = `apps/web`** 기준으로 `.next` 로 맞춤.)

```text
./
```

- 루트에 `pnpm-workspace.yaml`·`pnpm-lock.yaml`·`vercel.json`이 있고, `apps/web`은 `file:../../packages/schemas` 등 워크스페이스 패키지에 의존한다.
- 예전처럼 Root만 `apps/web`으로 두면 설치·빌드 시 상위 디렉터리가 없어 실패할 수 있다.
- 루트 `vercel.json`의 `installCommand` / `buildCommand`가 `pnpm install` 후 `apps/web`에서 Next 빌드를 수행한다.

로컬에서 `vercel deploy` 할 때도 **저장소 루트**에서 실행한다 (`apps/web` 폴더 안에서 실행하면 업로드 범위가 잘려 동일 오류가 난다).

## 필수 환경변수

### Supabase

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`

설명:
- `NEXT_PUBLIC_*` 는 브라우저에서도 사용
- `SUPABASE_SERVICE_ROLE_KEY` 는 서버 전용

**도서 카테고리·전체 목록·검색(`/books`, `/api/search`)**  
- 서버는 **Meilisearch**가 설정되어 있으면 그걸 우선 사용하고, 실패·0건이면 **Supabase `books` 테이블**로 폴백합니다.  
- **프로덕션**에서 Meilisearch를 안 쓰거나 인덱스가 비어 있으면, 폴백이 동작하려면 **`SUPABASE_SERVICE_ROLE_KEY`가 반드시 필요**합니다. 이 키가 없으면 카테고리 탭·목록이 비어 보일 수 있습니다.  
- 상단 검색창 **자동완성**도 Meilisearch가 없을 때는 같은 서비스 롤 키가 있으면 DB 폴백을 씁니다. (로컬에서만 `ALLOW_AUTOCOMPLETE_FIRESTORE_FALLBACK=false`로 끄는 경우는 그대로 끔.)
- **Vercel에는 `NEXT_PUBLIC_MEILISEARCH_HOST=http://localhost:7700` 을 넣지 마세요.** 서버 입장의 localhost라 연결되지 않고, 예전에는 `/api/search`가 CDN에 잠깐 캐시되며 빈 목록이 남을 수도 있었습니다. Meilisearch를 쓰려면 **클라우드/공인 URL**만 넣거나, 안 쓰면 Meili 관련 변수를 비워 두면 Supabase 폴백만 씁니다.
- `ALLOW_SEARCH_FIRESTORE_FALLBACK=false` 를 Vercel에 넣어 두면 서비스 롤이 있어도 **DB 폴백이 꺼집니다.** 넣었다면 제거하거나 `true`로 두세요.

#### 어드민 이미지·팝업 (Supabase Storage — 기본)

1. [Supabase 대시보드](https://supabase.com/dashboard) → 프로젝트 → **Storage** → **New bucket**
2. 이름 예: `cms-public` — **Public bucket** 을 켜기 (스토어·어드민 미리보기에서 URL로 바로 보이게)
3. Vercel 환경 변수: **`SUPABASE_STORAGE_BUCKET=cms-public`** (버킷 이름과 동일)

`SUPABASE_STORAGE_BUCKET` 이 있으면 `/api/admin/upload` 는 **Supabase Storage만** 쓴다(같은 요청에서 Firebase Storage는 호출하지 않음). 실패 시 로컬 `public/uploads` 폴백만 시도한다.

**스토어 표시:** 업로드 API가 돌려주는 **HTTPS URL**이 어드민 저장 시 Supabase `cms`(`key=home`) JSON의 `imageUrl` 등에 들어가고, 스토어는 그 문자열만 사용한다. 업로드 후 **저장(팝업 추가·배너 저장)** 까지 해야 반영된다.

### Firebase Client

- `NEXT_PUBLIC_FIREBASE_API_KEY`
- `NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN`
- `NEXT_PUBLIC_FIREBASE_PROJECT_ID`
- `NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET`
- `NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID`
- `NEXT_PUBLIC_FIREBASE_APP_ID`

설명:
- 로그인, 회원가입, 클라이언트 인증에 필요 (`NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET` 는 클라이언트 SDK 설정용일 뿐, 이미지 업로드는 Supabase Storage 사용 시 필수는 아님)

### Firebase Admin

- `FIREBASE_ADMIN_PROJECT_ID`
- `FIREBASE_ADMIN_CLIENT_EMAIL`
- `FIREBASE_ADMIN_PRIVATE_KEY`

설명:
- 서버에서 Firebase ID 토큰 검증, 어드민 권한 확인, 카카오 콜백 등 **인증·계정 API**에 필요 (파일 저장과 무관)

주의:
- `FIREBASE_ADMIN_PRIVATE_KEY` 는 Vercel에 넣을 때 줄바꿈 포함 값이어야 한다.
- 보통 `-----BEGIN PRIVATE KEY-----` 로 시작한다.

**업로드 503 (`STORAGE_UNAVAILABLE`)** — `SUPABASE_STORAGE_BUCKET` 을 쓰는 경우: 응답의 `detail`·`hint`는 **Supabase Storage** 기준으로 보면 된다. 버킷 공개 여부·이름·정책을 확인한다.

**레거시:** `SUPABASE_STORAGE_BUCKET` 을 **비워 두었을 때만** `/api/admin/upload` 가 Firebase Storage를 시도한다. 그때 버킷 404·IAM 오류는 GCP/Firebase 콘솔에서 본다.

스토어에서 `/uploads/banners_…` 가 **404**이면 로컬 개발 때 저장된 경로다. 프로덕션에서는 **다시 업로드**해 Supabase(또는 쓰는 스토리지)의 **전체 HTTPS URL**이 CMS에 들어가게 한다.

### 사이트 URL

- `NEXT_PUBLIC_SITE_URL`

값 예시:

```text
https://your-project-name.vercel.app
```

설명:
- sitemap
- robots
- metadata base URL

## Vercel에서 특히 주의할 점

### 1. `.env.local` 은 자동 반영되지 않음

- 로컬 `.env.local` 값이 Vercel에 자동 업로드되지는 않는다.
- Environment Variables에 직접 입력해야 한다.

### 2. service role key 노출 금지

- `SUPABASE_SERVICE_ROLE_KEY` 는 절대 `NEXT_PUBLIC_` 로 만들지 않는다.
- 브라우저 노출 금지

### 3. Firebase Admin private key 형식

- 값이 깨지면 서버에서 인증 검증이 실패한다.
- 배포 후 로그인은 되는데 API가 401/500이면 이 값을 먼저 확인한다.

## 배포 후 점검 기준

- 페이지가 500 없이 열린다
- Supabase 데이터가 실제로 보인다
- 로그인/회원가입이 된다
- 관리자 페이지가 열린다
- 책 목록/검색/콘서트/콘텐츠가 보인다

## 문제 발생 시 우선 점검 순서

1. Vercel Deployment Logs
2. Vercel Runtime Logs
3. Environment Variables 누락 여부
4. Supabase URL / key 오입력 여부
5. Firebase Admin private key 형식
6. `NEXT_PUBLIC_SITE_URL` 값
