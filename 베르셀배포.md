# 베르셀 배포

## 목적

- 지금 단계에서는 도메인 연결 없이 `Vercel`만 먼저 배포한다.
- 배포 주소는 기본 `*.vercel.app` 도메인을 사용한다.
- 추후 가비아 도메인과 Cloudflare는 별도 작업으로 연결한다.

## 배포 전 상태

- `apps/web`는 Supabase 기반으로 동작한다.
- Firebase는 Auth, Admin Auth, Storage 용도로만 사용한다.
- Firestore 주요 데이터는 Supabase로 이관 완료 상태다.

## 배포 순서

1. 현재 저장소를 GitHub에 push
2. Vercel에서 `New Project`
3. GitHub 저장소 import
4. 프로젝트 설정
   - Framework Preset: `Next.js`
   - Root Directory: `apps/web`
5. Environment Variables 입력
6. Deploy

## Root Directory

반드시 아래처럼 설정한다.

```text
apps/web
```

이 프로젝트는 모노레포 구조이므로 루트를 잘못 잡으면 빌드가 어긋난다.

## 필수 환경변수

### Supabase

- `NEXT_PUBLIC_SUPABASE_URL` — **반드시 클라우드 URL** (`https://….supabase.co`). 로컬 `http://127.0.0.1:54321` 을 Vercel에 넣으면 프로덕션에서 DB 연결 실패 → **도서 목록·검색 0건**
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`

설명:
- `NEXT_PUBLIC_*` 는 브라우저에서도 사용
- `SUPABASE_SERVICE_ROLE_KEY` 는 서버 전용

### Firebase Client

- `NEXT_PUBLIC_FIREBASE_API_KEY`
- `NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN`
- `NEXT_PUBLIC_FIREBASE_PROJECT_ID`
- `NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET`
- `NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID`
- `NEXT_PUBLIC_FIREBASE_APP_ID`

설명:
- 로그인, 회원가입, 클라이언트 인증, Storage 접근에 필요

### Firebase Admin

- `FIREBASE_ADMIN_PROJECT_ID`
- `FIREBASE_ADMIN_CLIENT_EMAIL`
- `FIREBASE_ADMIN_PRIVATE_KEY`
- (선택) `FIREBASE_SERVICE_ACCOUNT_JSON` — 서비스 계정 JSON **전체**를 한 변수에 넣기(Vercel에서 `private_key` 줄바꿈 깨짐 방지). 설정 시 위 3개보다 우선
- (선택) `FIREBASE_STORAGE_BUCKET` — 서버 업로드 전용. 미설정 시 `NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET` 또는 `프로젝트ID.firebasestorage.app` 추정

설명:
- 서버에서 Firebase ID 토큰 검증
- 관리자 권한 확인
- Storage 서버 처리

주의:
- `FIREBASE_ADMIN_PRIVATE_KEY` 는 Vercel에 넣을 때 줄바꿈 포함 값이어야 한다.
- 보통 `-----BEGIN PRIVATE KEY-----` 로 시작한다.

어드민 이미지·팝업 업로드(`/api/admin/upload`)는 **Firebase Storage**에 올라간다. Vercel 서버 디스크에는 쓸 수 없으므로 로컬 `public/uploads` 폴백은 프로덕션에서 동작하지 않는다. 반드시 위 Admin 키가 맞고, Firebase 콘솔에서 Storage가 켜져 있으며, 버킷명이 콘솔과 일치해야 한다(`FIREBASE_STORAGE_BUCKET` 또는 `NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET`; 비어 있으면 코드가 `프로젝트ID.firebasestorage.app` 으로 추정).

**권한:** Google Cloud Console → IAM에서 `firebase-adminsdk-…@….iam.gserviceaccount.com` 에 **Storage Object Admin**(`roles/storage.objectAdmin`) 또는 상위 역할이 있어야 업로드가 된다. 503 응답 JSON의 `storageError` 필드에 GCS 오류 코드·메시지가 포함되면 그에 맞춰 수정하면 된다.

### 사이트 URL

## Vercel에서 특히 주의할 점

### 1. `.env.local` 은 자동 반영되지 않음

- 로컬 `.env.local` 값이 Vercel에 자동 업로드되지는 않는다.
- Environment Variables에 직접 입력해야 한다.

### 2. service role key 노출 금지

- `SUPABASE_SERVICE_ROLE_KEY` 는 절대 `NEXT_PUBLIC_` 로 만들지 않는다.
- 브라우저 노출 금지

### 3. Firebase Admin private key 형식

- 값이 깨지면 서버에서 인증 검증이 실패한다.
- 배포 후 로그인은 되는데 API가 401/500이면 이 값을 먼저 확인한다.

## 배포 후 확인 URL

스토어:

- `/`
- `/books`
- `/concerts`
- `/content`
- `/search?q=책`

인증:

- `/login`
- `/signup`

관리자:

- `/admin`
- `/admin/books`
- `/admin/orders`
- `/admin/cms`
- `/admin/settings`

API:

- `/api/store/cms-live`
- `/api/search?q=책`
- `/api/admin/dashboard`

## 배포 후 점검 기준

- 페이지가 500 없이 열린다
- Supabase 데이터가 실제로 보인다
- 로그인/회원가입이 된다
- 관리자 페이지가 열린다
- 책 목록/검색/콘서트/콘텐츠가 보인다

## 문제 발생 시 우선 점검 순서

1. Vercel Deployment Logs
2. Vercel Runtime Logs
3. Environment Variables 누락 여부
4. Supabase URL / key 오입력 여부
5. Firebase Admin private key 형식
6. `NEXT_PUBLIC_SITE_URL` 값

## 자주 겹치는 증상 (503 업로드 / 이미지 404 / 도서 0건)

| 증상 | 흔한 원인 | 조치 |
|------|-----------|------|
| 어드민 업로드 **503** | Firebase Admin env 누락·깨짐 → Storage 실패 후 로컬 디스크 폴백 시도 → Vercel은 쓰기 불가 | `FIREBASE_ADMIN_*` 또는 `FIREBASE_SERVICE_ACCOUNT_JSON` 을 Production에 정확히 입력 후 **Redeploy** |
| `/uploads/...` **404** | 예전에 로컬에만 있던 파일 URL이 프로덕션에 없음 | 어드민에서 해당 배너·팝업 등을 **다시 업로드**해 Storage URL로 교체 |
| `/books` **전체 0건** | `NEXT_PUBLIC_SUPABASE_URL` 이 로컬(`127.0.0.1:54321`)이거나 키 불일치 | Supabase 대시보드의 **클라우드** URL·`anon`·`service_role` 키로 교체 후 **Redeploy** |

환경 변수 수정만으로는 이전 빌드가 남을 수 있으므로, 저장 후 **Deployments → Redeploy** 로 한 번 올리는 것을 권장한다.
