# 베르셀 배포

## 목적

- 지금 단계에서는 도메인 연결 없이 `Vercel`만 먼저 배포한다.
- 배포 주소는 기본 `*.vercel.app` 도메인을 사용한다.
- 추후 가비아 도메인과 Cloudflare는 별도 작업으로 연결한다.

## 배포 전 상태

- `apps/web`는 Supabase 기반으로 동작한다.
- Firebase는 Auth, Admin Auth, Storage 용도로만 사용한다.
- Firestore 주요 데이터는 Supabase로 이관 완료 상태다.

## 배포 순서

1. 현재 저장소를 GitHub에 push
2. Vercel에서 `New Project`
3. GitHub 저장소 import
4. 프로젝트 설정
   - Framework Preset: `Next.js`
   - Root Directory: `apps/web`
5. Environment Variables 입력
6. Deploy

## Root Directory

반드시 아래처럼 설정한다.

```text
apps/web
```

이 프로젝트는 모노레포 구조이므로 루트를 잘못 잡으면 빌드가 어긋난다.

## 필수 환경변수

### Supabase

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`

설명:
- `NEXT_PUBLIC_*` 는 브라우저에서도 사용
- `SUPABASE_SERVICE_ROLE_KEY` 는 서버 전용

### Firebase Client

- `NEXT_PUBLIC_FIREBASE_API_KEY`
- `NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN`
- `NEXT_PUBLIC_FIREBASE_PROJECT_ID`
- `NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET`
- `NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID`
- `NEXT_PUBLIC_FIREBASE_APP_ID`

설명:
- 로그인, 회원가입, 클라이언트 인증, Storage 접근에 필요

### Firebase Admin

- `FIREBASE_ADMIN_PROJECT_ID`
- `FIREBASE_ADMIN_CLIENT_EMAIL`
- `FIREBASE_ADMIN_PRIVATE_KEY`

설명:
- 서버에서 Firebase ID 토큰 검증
- 관리자 권한 확인
- Storage 서버 처리

주의:
- `FIREBASE_ADMIN_PRIVATE_KEY` 는 Vercel에 넣을 때 줄바꿈 포함 값이어야 한다.
- 보통 `-----BEGIN PRIVATE KEY-----` 로 시작한다.

어드민 이미지·팝업 업로드(`/api/admin/upload`)는 **Firebase Storage**에 올라간다. Vercel 서버 디스크에는 쓸 수 없으므로 로컬 `public/uploads` 폴백은 프로덕션에서 동작하지 않는다. 반드시 위 Admin 키가 맞고, Firebase 콘솔에서 Storage가 켜져 있으며, `NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET` 이 콘솔 기본 버킷과 같아야 한다(비어 있으면 코드가 `프로젝트ID.firebasestorage.app` 으로 추정한다).

503이 나오면 응답 JSON의 `detail`에 GCS/Firebase 실제 오류 메시지가 포함된다. 여전히 실패할 때는 **GCP 콘솔 → IAM**에서 `firebase-adminsdk-…@…iam.gserviceaccount.com` 계정에 **Storage 객체 관리자**(또는 **Storage 관리자**) 역할이 있는지 확인한다. `FIREBASE_ADMIN_PROJECT_ID`는 서비스 계정 JSON의 `project_id`와 동일해야 한다.

**`404: The specified bucket does not exist`**: Vercel의 `NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET` 값이 Firebase에 실제로 만들어진 기본 버킷과 다를 때 난다. Firebase 콘솔 → **Storage** 또는 **프로젝트 설정**에서 표시되는 **정확한 버킷 이름**을 복사해 넣는다. 구 프로젝트는 `프로젝트ID.appspot.com` 만 있을 수 있고, 안내 문구는 `프로젝트ID.firebasestorage.app` 일 수 있어 둘이 다르면 콘솔 값을 따른다. (코드는 두 접미사 후보를 순서대로 `exists` 검사해 살아 있는 버킷을 고른다.)

스토어에서 `/uploads/banners_…` 같은 경로가 **404**이면, 예전에 로컬 폴백으로 저장된 URL이라 프로덕션에는 파일이 없다. 어드민에서 해당 배너/이미지를 **다시 업로드**해 Firebase URL로 바꾼다.

### 사이트 URL

- `NEXT_PUBLIC_SITE_URL`

값 예시:

```text
https://your-project-name.vercel.app
```

설명:
- sitemap
- robots
- metadata base URL

## Vercel에서 특히 주의할 점

### 1. `.env.local` 은 자동 반영되지 않음

- 로컬 `.env.local` 값이 Vercel에 자동 업로드되지는 않는다.
- Environment Variables에 직접 입력해야 한다.

### 2. service role key 노출 금지

- `SUPABASE_SERVICE_ROLE_KEY` 는 절대 `NEXT_PUBLIC_` 로 만들지 않는다.
- 브라우저 노출 금지

### 3. Firebase Admin private key 형식

- 값이 깨지면 서버에서 인증 검증이 실패한다.
- 배포 후 로그인은 되는데 API가 401/500이면 이 값을 먼저 확인한다.

## 배포 후 확인 URL

스토어:

- `/`
- `/books`
- `/concerts`
- `/content`
- `/search?q=책`

인증:

- `/login`
- `/signup`

관리자:

- `/admin`
- `/admin/books`
- `/admin/orders`
- `/admin/cms`
- `/admin/settings`

API:

- `/api/store/cms-live`
- `/api/search?q=책`
- `/api/admin/dashboard`

## 배포 후 점검 기준

- 페이지가 500 없이 열린다
- Supabase 데이터가 실제로 보인다
- 로그인/회원가입이 된다
- 관리자 페이지가 열린다
- 책 목록/검색/콘서트/콘텐츠가 보인다

## 문제 발생 시 우선 점검 순서

1. Vercel Deployment Logs
2. Vercel Runtime Logs
3. Environment Variables 누락 여부
4. Supabase URL / key 오입력 여부
5. Firebase Admin private key 형식
6. `NEXT_PUBLIC_SITE_URL` 값

## 지금 단계의 권장 운영 방식

- 먼저 `Vercel`만 연결해서 `vercel.app` 주소로 운영
- 기능 안정화 확인

- 그 다음 Cloudflare DNS 연결
