## 1. 개요 (Executive Summary)
본 문서는 10년 차 이상의 시니어 개발자 및 사업가 관점에서 시스템을 전수 감사(Audit)한 결과입니다. 단순한 기능 유무를 넘어, **상용화 시 서비스 장애를 유발할 수 있는 기술적 부채**와 **매출 직결되는 비즈니스 리스크**를 모두 포함하고 있습니다. 본 문서는 일괄 수정 작업을 위한 최종 설계 원본입니다.

## 2. [심층 감사] 기술적 결함 및 서비스 리스크 (Critical Risks)

### 2-1. 리뷰 시스템 장애 리스크 (Scalability Failure)
*   **문제점:** `createReview` CF에서 평점 집계 시 해당 도서의 **모든 리뷰를 전수 조회**하여 트랜잭션 내에서 계산함.
*   **리스크:** 베스트셀러처럼 리뷰가 500건 이상 쌓이는 경우, 리뷰가 작성될 때마다Firestore 읽기 비용이 폭증하며 트랜잭션 타임아웃으로 인해 리뷰 작성이 불가능해짐.
*   **수정 방안:** `books` 컬렉션에 `totalRatingSum` 필드를 추가하여, 매번 전수 조사하지 않고 `(기존 합계 + 신규 점수) / (기존 개수 + 1)` 방식으로 O(1) 연산 리팩토링 필수.

### 2-2. 이벤트 신청 동시성 리스크 (Overselling Race Condition)
*   **문제점:** 정원 초과 체크(`registeredCount >= capacity`)가 트랜잭션 외부(Read)에서 수행됨.
*   **리스크:** 인기 강연 등 신청이 몰리는 시점에 다수의 사용자가 동시에 체크를 통과하여 **정원을 초과한 중복 신청**이 발생하는 '오버셀링' 리스크.
*   **수정 방안:** 정원 확인 및 카운트 증가 로직을 `db.runTransaction` 내부로 완전히 포함시켜 원자성(Atomicity) 확보.

### 2-3. 주문 관리 및 물류 데이터 단절
*   **치명적 결함:** **주문 내역 엑셀 다운로드 기능 부재**.
*   **사업적 리스크:** 하루 주문이 10건만 넘어가도 관리자가 수기로 송장 데이터를 택배 시스템에 입력해야 함. 운영 효율 0점.
*   **수정 방안:** `admin/orders` 내 기간별/상태별 주문 데이터를 택배사 양식에 맞춘 CSV/Excel 추출 기능 긴급 도입.

## 3. [검색 및 큐레이션] 사업적 확장성 보완

### 3-1. 하이브리드 검색 및 사용자 이탈 방지
*   **문제점:** 내부 DB 결과가 0건일 때 대안 부재.
*   **수정 방안:** 내부 DB miss 시 알라딘 API 자동 연동. 단, 우리 DB에 있는 책은 중복 노출되지 않도록 **ISBN 기반 Deduplication** 프로세스 적용 필수.
*   **리드 수집:** 없는 도서에 대해 '입고 알림 신청(Lead Gen)' 연동하여 잠재 고객 확보.

### 3-2. 독립서점 감성 UX (Curation 2.0)
*   **문제점:** 큐레이션 도서가 단순 나열형(Carousel)으로만 출력됨.
*   **수정 방안:** 큐레이션 상세 작성 시 **Markdown 기반의 장문 추천사** 지원. 텍스트와 도서가 잡지처럼 배치되는 감성 레이아웃 컴포넌트 추가.

## 6. 추가 감사 A - 거짓 완료 항목 목록 (False Completion Audit)

아래 항목들은 "화면이 존재한다", "관리자에서 저장된다", "API가 있다"는 이유로 완료처럼 보이지만, 실제 PRD 기준의 최종 사용자 동작이나 운영 흐름까지 완성되지 않아 **완료로 간주하면 안 되는 항목들**이다.

### 6-1. 배너/팝업 관리
- **현재 상태:** `apps/web/src/app/(admin)/admin/marketing/page.tsx`에서 배너와 팝업 데이터를 저장할 수 있다.
- **문제점:** 배너는 홈에서 일부 소비되지만, 팝업은 스토어 레이아웃(`apps/web/src/app/(store)/layout.tsx`)에서 아예 읽지도 않고 렌더링도 하지 않는다.
- **판정:** **부분 완료**. PRD 기준으로는 "운영자가 등록하면 스토어에 노출"이므로 팝업은 미완료다.

### 6-2. CMS 큐레이션
- **현재 상태:** `apps/web/src/app/(admin)/admin/cms/page.tsx`에서 추천 도서, 이달의 책, 테마 큐레이션을 입력/저장할 수 있다.
- **문제점 1:** 관리자 저장 구조는 `themeCurations[].books`인데, 공개 스토어 로직 `apps/web/src/lib/cmsHome.ts`는 `themeCurations[].isbns`를 기대한다.
- **문제점 2:** recommendationText/description 등 운영자가 입력한 설명형 콘텐츠가 공개 페이지에서 충분히 소비되지 않는다.
- **판정:** **겉보기 완료 / 실제 미완료**. 저장 UI가 있어도 스토어 계약이 깨져 있으면 완료가 아니다.

### 6-3. 콘텐츠 관리
- **현재 상태:** `apps/web/src/app/(admin)/admin/content/page.tsx`에서 "본문(마크다운)"을 입력할 수 있다.
- **문제점:** 공개 페이지 `apps/web/src/app/(store)/content/[slug]/page.tsx`는 마크다운 렌더링이 아니라 문자열 그대로 출력한다.
- **판정:** **기능 과장 완료**. 작성 UX와 공개 렌더링 결과가 다르므로 PRD 품질 기준 미달이다.

### 6-4. 이벤트 신청
- **현재 상태:** 이벤트 신청 UI, 관리 UI, 참가자 조회까지 존재한다.
- **문제점:** `functions/src/events/registerEvent.ts`에서 정원 초과 체크가 트랜잭션 외부에 있어 동시성 상황에서 초과 신청이 발생할 수 있다.
- **판정:** **기능 존재 / 운영 품질 미달**. 인기 이벤트에서 실서비스 장애를 유발할 수 있다.

### 6-5. 리뷰 시스템
- **현재 상태:** 구매자만 리뷰 작성, 중복 작성 방지, 평균 평점 갱신 흐름이 있다.
- **문제점:** `functions/src/review/createReview.ts`가 새 리뷰 작성 때마다 동일 도서의 모든 리뷰를 다시 읽는다.
- **판정:** **초기 데모 수준 완료**. 트래픽/리뷰가 증가하면 바로 확장성 문제가 발생한다.

### 6-6. 주문 관리
- **현재 상태:** `apps/web/src/app/(admin)/admin/orders/page.tsx`에서 주문 목록, 상태 필터, 배송 상태 변경, 반품 완료 버튼이 있다.
- **문제점:** PRD 및 운영 현실 기준의 핵심인 CSV/Excel 추출, 택배사 전달 양식, 반품 정산/재고 복원 정합성, 기간 기반 운영 기능이 없다.
- **판정:** **화면형 관리툴** 수준이며, 실운영용 주문 관리 시스템으로 보기 어렵다.

### 6-7. 검색
- **현재 상태:** 검색창, 자동완성, 내부 검색 API, Meilisearch 우선 검색이 있다.
- **문제점:** PRD가 요구한 DB miss 시 알라딘 API fallback, ISBN dedup, 입고 알림 Lead 수집이 없다.
- **판정:** **기본 검색만 구현된 상태**. PRD에서 말한 "사업적 검색"은 아직 미완성이다.

### 6-8. 업로드 시스템
- **현재 상태:** `ImagePreviewUploader`와 관리자 업로드 API가 있다.
- **문제점:** 실제 운영 중 `POST /api/admin/upload 500`이 발생했고, 이미지 업로드가 막히는 상태였다.
- **판정:** **구현 존재 != 완료**. 실제 업로드 성공이 검증되지 않으면 관리자 CMS의 핵심 가치가 무너진다.

## 7. 추가 감사 B - 즉시 수정해야 할 항목 (Immediate Fix Queue)

아래 항목들은 단순 개선이 아니라, 현재 상태에서 **서비스 체감 품질 저하**, **운영 불가**, **데이터 정합성 위험**, **상용화 차단**을 발생시키는 항목들이다. 우선순위는 반드시 아래 순서를 따른다.

### 7-1. P0 - 스토어 노출이 끊긴 CMS 항목 복구
1. **팝업 스토어 노출 구현**
   - 대상: `apps/web/src/app/(store)/layout.tsx`
   - 요구사항:
     - `cms/home.popup` 읽기
     - 활성 여부 / 종료일 체크
     - 반응형 팝업 레이어 렌더링
     - "오늘 보지 않기" 쿠키 처리
   - 이유: 관리자에서 저장해도 사용자에게 안 보이면 기능 자체가 성립하지 않는다.

2. **테마 큐레이션 데이터 구조 통일**
   - 대상: `apps/web/src/app/(admin)/admin/cms/page.tsx`, `apps/web/src/lib/cmsHome.ts`, `apps/web/src/app/(store)/page.tsx`, `apps/web/src/app/(store)/curation/[id]/page.tsx`
   - 요구사항:
     - 관리자 저장 구조와 공개 소비 구조를 하나로 통일
     - 저장 즉시 홈/큐레이션에서 정상 노출 검증
   - 이유: 현재는 저장과 소비 계약이 달라 실제 노출이 깨질 수 있다.

3. **이미지 업로드 500 문제 완전 해결**
   - 대상: `apps/web/src/components/admin/ImagePreviewUploader.tsx`, `apps/web/src/app/api/admin/upload/route.ts`, Firebase Storage 설정
   - 요구사항:
     - 배너/팝업/콘텐츠/이벤트 이미지 업로드 성공 검증
     - 저장 URL이 공개 페이지에서 실제 렌더링되는지 검증
   - 이유: 업로드가 막히면 CMS 운영 자체가 중단된다.

### 7-2. P1 - 실서비스 장애를 막는 로직 수정
1. **이벤트 정원 체크 트랜잭션 내부화**
   - 대상: `functions/src/events/registerEvent.ts`
   - 요구사항:
     - 정원 확인, 중복 체크, 카운트 증가를 동일 트랜잭션 문맥으로 이동
   - 이유: 인기 이벤트 오버셀링 방지.

2. **리뷰 집계 O(1) 리팩토링**
   - 대상: `functions/src/review/createReview.ts`, `books` 스키마
   - 요구사항:
     - `totalRatingSum` 또는 동등한 누적 필드 추가
     - 전체 리뷰 재조회 제거
   - 이유: 리뷰 수 증가 시 비용 폭증과 트랜잭션 실패 위험 제거.

3. **결제/취소/웹훅 정합성 점검**
   - 대상: 결제 취소, 웹훅, 재고 복원 관련 로직 전체
   - 요구사항:
     - 취소/재고복원 중복 수행 여부 검증
     - 상태 이력 및 보상 처리 기준 명확화
   - 이유: 재고와 주문 상태가 한 번만 정확히 갱신되어야 한다.

### 7-3. P2 - 운영형 MVP 기준 미달 항목 보완
1. **주문 CSV/Excel 추출**
   - 대상: `apps/web/src/app/(admin)/admin/orders/page.tsx`, 관련 API
   - 요구사항:
     - 기간/상태 필터
     - 택배사 전달용 CSV/Excel export
   - 이유: 수기 입력 운영은 MVP도 아니다.

2. **콘텐츠 마크다운 실제 렌더링**
   - 대상: `apps/web/src/app/(store)/content/[slug]/page.tsx`
   - 요구사항:
     - 관리자 입력 포맷과 공개 렌더링 포맷 일치
   - 이유: 작성-노출 계약 일치 필요.

3. **하이브리드 검색 도입**
   - 대상: `apps/web/src/app/api/search/route.ts`
   - 요구사항:
     - 내부 DB miss 시 알라딘 fallback
     - ISBN dedup
     - 입고 알림 리드 수집
   - 이유: 검색 실패는 곧 이탈이다.

4. **도서 상세 Product JSON-LD 추가**
   - 대상: `apps/web/src/app/(store)/books/[slug]/page.tsx`
   - 요구사항:
     - 가격, 재고, 평점 포함 Product Schema 삽입
   - 이유: PRD의 SEO 요구 직접 반영.

## 8. 문서 해석 원칙 (중요)

본 추가 섹션은 기존 `수정prd` 내용을 교체하거나 축소하지 않는다.  
오히려 현재 구현 상태를 기준으로, "무엇이 진짜 완료이고 무엇이 완료처럼 보이는 미완료인지"를 더 명확하게 식별하기 위한 운영 보조 지침이다.

즉, 앞으로의 수정 우선순위는 아래 원칙을 따른다.

1. **저장되면 끝이 아니라, 스토어에 노출되어야 완료다.**
2. **화면이 있으면 끝이 아니라, 데이터 계약이 맞아야 완료다.**
3. **단일 사용자 테스트가 되면 끝이 아니라, 동시성/운영 리스크가 없을 때 완료다.**
4. **관리자 입력 포맷과 공개 렌더링 결과가 일치해야 완료다.**

## 9. 추가 감사 C - docs_PRD 기준 실제 구현 재검수

### 9-4. 고위험 불일치 (High-Risk Mismatches)

#### 9-4-1. 테마 큐레이션 데이터 계약 불일치
- 관리자 CMS는 `themeCurations[].books` 형태로 저장하는 반면,
- `apps/web/src/lib/cmsHome.ts`와 스토어는 `themeCurations[].isbns`를 기대한다.
- 저장과 소비 계약이 달라 홈/큐레이션 페이지가 깨질 수 있다.

#### 9-4-2. 배너/팝업 데이터 계약 불일치
- 관리자는 `position`, `popup`까지 저장하지만,
- 스토어는 `heroBanners`의 일부만 단일 캐러셀로 사용한다.
- PRD의 "운영자가 등록하면 해당 영역에 노출" 계약이 실제 코드에서 지켜지지 않는다.

#### 9-4-3. 이벤트 정원 동시성 문제
- `functions/src/events/registerEvent.ts`는 정원 초과 체크가 트랜잭션 외부에 있다.
- PRD의 운영 안정성 기준과 충돌한다.

#### 9-4-4. 리뷰 집계 확장성 문제
- `functions/src/review/createReview.ts`는 새 리뷰 작성 시 전체 리뷰를 다시 읽는다.
- PRD가 요구하는 상용 트래픽 대응 관점에서 위험하다.

#### 9-4-5. 관리자 전용 기능의 권한 경계 약화
- `functions/src/order/bulkCreateBooks.ts`는 UI는 관리자 화면 전용이지만 함수 단에서 admin claim 검증이 강하게 보장되지 않는다.
- 운영/보안 리스크가 있다.

#### 9-4-6. 콘텐츠 작성 포맷과 노출 포맷 불일치
- 관리자는 마크다운 기반으로 인식하고 작성하지만,
- 공개 페이지는 마크다운 렌더링 없이 plain text 출력이다.
- CMS 운영자 기대와 사용자 노출 결과가 다르다.

#### 9-4-7. 업로드 시스템의 실동작 불안정
- 업로드 UI/API는 존재하나 실제 운영 중 `POST /api/admin/upload 500`이 발생했다.
- 구현 존재 여부보다 실제 성공률과 안정성 검증이 더 중요하다.

## 10. 재검수 결론

`docs_PRD` 기준으로 볼 때 현재 구현은 **핵심 뼈대는 상당수 존재하지만**, 다음 영역은 아직 PRD 충족으로 보기 어렵다.

1. **운영자가 입력한 데이터가 스토어에 실제 반영되는 CMS/마케팅 흐름**
2. **데이터 계약이 맞는 큐레이션 시스템**
3. **운영 가능한 주문/물류 기능**
4. **상용 수준의 검색 확장성**
5. **SEO/분석/정합성 보완**

따라서 앞으로의 수정은 "화면 추가"보다 아래 원칙으로 진행해야 한다.

- 관리자 저장 → 스토어 노출까지 완결될 것
- 저장 구조와 소비 구조가 동일할 것
- 동시성/운영 리스크를 먼저 제거할 것
- PRD에 있는 항목은 '기능 있음'이 아니라 '실제 동작함' 기준으로 완료 판정할 것

   - 홈 소개 영역의 CMS화가 빠져 있었다.
5. **인기 검색어 / 검색 분석**
   - 검색 확장성 보완 항목에 포함되어야 한다.
6. **GA4 / Vercel Analytics**
   - SEO/운영 분석 축으로 별도 작업화가 필요하다.
7. **관리자 권한 경계 강화**
   - `bulkCreateBooks` 등 관리자 전용 로직 보호가 백로그에 충분히 드러나지 않았다.
8. **실제 엑셀(.xlsx/.xls) 지원 여부 확정**
   - 지원할지, CSV 전용으로 문서/화면을 정정할지 결정이 필요하다.
9. **대시보드 KPI 고도화**
   - 원문 PRD에는 있으나 후반 우선순위 문서에서 상대적으로 약하게 다뤄졌다.

### 11-3. 의존성 충돌 정리

아래 작업들은 순서를 잘못 잡으면 재작업이 발생한다.

1. **업로드 안정화 이전에 마케팅/CMS 확장은 금지**
   - 업로드가 불안정하면 배너/팝업/콘텐츠/이벤트 개선 검증이 불가능하다.
2. **큐레이션 데이터 계약 통일 이전에 큐레이션 UI 확장은 금지**
   - 저장/소비 구조가 다르면 표현 개선은 모두 임시 작업이 된다.
3. **결제 무결성 점검 이전에 반품/환불 운영 확장 금지**
   - 상태 전이 규칙이 고정되지 않으면 운영 기능이 이중 처리 위험을 키운다.
4. **검색 fallback 설계 이전에 검색 분석 확장 금지**
   - 검색 결과 구조가 변하기 전 analytics 이벤트를 먼저 심으면 다시 고쳐야 한다.

## 12. 통합 실행 백로그 (Master Execution Backlog)

## 15. [시니어 감사 추가] 상용화 직전 최종 점검 및 고도화 (Final Polish & Audit)

본 섹션은 앞서 진행된 보완 작업을 바탕으로, 10년 차 이상의 시니어 개발자 관점에서 상용 서비스를 위해 반드시 넘어야 할 **'마지막 2%'의 기술적/비즈니스적 임계점**을 정리한 내용입니다.

### 15-1. 기술적 결함 보완 (Critical Fixes)

#### 15-1-1. 리뷰 평점(Rating) 정합성 즉시 동기화
*   **리스크:** 현재 `ratingTotal`과 `reviewCount`만 갱신되고 있어, 정렬에 사용되는 `rating`(평균점수) 필드가 즉시 업데이트되지 않을 경우 평점순 검색 결과가 왜곡됨.
*   **수정 사항:** `createReview` 트랜잭션 내에서 `rating = (기존 합계 + 신규 점수) / (기존 개수 + 1)`을 계산하여 **`rating` 필드를 직접 갱신**하도록 보완.

#### 15-1-2. 주문 엑셀 추출 확장성 개선
*   **리스크:** 현재 500건을 전수 조회하여 메모리에서 필터링하는 방식은 주문량 증가 시 성능 저하 및 메모리 초과 위험이 있음.
*   **수정 사항:** Firestore Native Query(`where` 문)를 사용하여 **DB 단에서 기간/상태 필터링**을 수행하도록 API 최적화.

#### 15-1-3. 하이브리드 검색 회복력(Resilience) 강화
*   **리스크:** 외부 알라딘 API의 응답 지연이나 장애가 발생할 경우 우리 서비스의 검색 기능이 함께 마비될 수 있음.
*   **수정 사항:** 알라딘 API 호출 시 **`AbortController` 기반 타임아웃(3초)** 설정 및 장애 시 빈 결과값을 즉시 반환하는 서킷 브레이커 로직 적용.

#### 15-1-4. 이미지 업로드 경로 보안 강화
*   **리스크:** 클라이언트가 `storagePath`를 임의로 지정할 수 있어 파일 덮어쓰기나 경로 조작 공격에 취약함.
*   **수정 사항:** 서버측에서 허용된 폴더(banners, popups, contents 등) **화이트리스트 검증**을 수행하고, 파일명을 서버에서 강제로 난수화하여 충돌 및 조작 방지.

### 15-2. SEO 및 비즈니스 UX 고도화 (Growth & UX)

#### 15-2-1. Google Rich Snippets (JSON-LD) 완전 도입
*   **수정 사항:** 도서 상세 페이지에 `schema.org/Product` 구조화 데이터를 삽입.
    - 필수 포함 필드: `name`(도서명), `image`(표지), `description`(설명), `isbn`, `offers`(`price`, `priceCurrency`, `availability`), `aggregateRating`(평균평점, 리뷰수).
*   **기대 효과:** 구글 검색 결과에서 별점과 가격이 노출되어 **유입률(CTR) 30% 이상 상승** 기대.

#### 15-2-2. 팝업 시스템 운영 고도화
*   **수정 사항:** 다중 팝업 노출 시 UI 겹침을 방지하기 위해 **우선순위(Priority)** 기반의 단일 노출 또는 스택형 UI 로직 보강. "오늘 보지 않기"의 서버측 쿠키 연동 옵션 검토.

#### 15-2-3. 운영 데이터 수집 강화 (Event Cancellation)
*   **수정 사항:** 이벤트 신청 취소 시 단순 삭제/상태 변경을 넘어 **'취소 사유'를 수집**하는 필드 추가.
*   **사업적 활용:** 반복적인 노쇼(No-show) 고객 및 서비스 불만 요인을 데이터로 관리하여 운영 정책 개선 자료로 활용.
